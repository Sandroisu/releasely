# v0.1.0 external validation — three-times-a-day

External validation of the published Releasely v0.1.0 GitHub Action against a real
Kotlin Multiplatform / Android project: `sandroisu/three-times-a-day`.

## Scope

- **Validated repository:** `sandroisu/three-times-a-day` — a real KMP/Android app
  (`androidApp` application module + `shared` KMP module). Referenced by repository
  slug only; no local absolute paths are recorded here.
- **Releasely version:** v0.1.0 (consumed through the `v0` major alias).
- **Action:** `Releasely Android Release Audit`.

## Onboarding experience

Wiring the Action into the project was simple: a single workflow step referencing
the `v0` alias, with default inputs. No project-specific configuration, no scanner
tuning, and no changes to the target repository were required to get a first audit.

## Initial audit

The first audit produced two findings:

| Severity | Finding |
| --- | --- |
| MEDIUM | Exported launcher `MainActivity`. |
| HIGH | Exported `MedicationReminderBootReceiver`. |

### BootReceiver remediation

`MedicationReminderBootReceiver` only ever handled the system `BOOT_COMPLETED`
broadcast and already re-checked the action inside `onReceive`. It did not need to be
exported to other apps, so `android:exported` was changed from `true` to `false`.

## Re-audit after remediation

After the receiver fix:

- **HIGH findings:** 0
- **MEDIUM findings:** 1 (the exported launcher `MainActivity`)

### Threshold behaviour

- With `fail-on: HIGH`, the job **succeeds** (no HIGH findings remain).
- With `fail-on: MEDIUM`, the job **fails with exit code 1** (the one MEDIUM finding
  reaches the threshold).
- The **Job Summary**, `report.md`, and `report.json` are all produced and uploaded
  **even when the threshold failure occurs** — the reports and artifacts survive a
  non-zero exit.

## Accuracy assessment

- **0 confirmed false positives.** Every finding pointed at a real manifest/config
  fact.
- The **launcher `Activity`** finding is *correct but low-value*: the main launcher
  activity is exported by necessity, so flagging it is technically accurate but rarely
  actionable. Recorded as an accepted finding.

## Observations to improve

1. **Overly generic recommendation for exported components.** The exported-component
   recommendation is the same broad "verify this must be exported…" text regardless of
   component kind, which is not very actionable for an unavoidable launcher activity.
2. **Inflated HIGH severity for a system-only BootReceiver.** A receiver that only
   accepts the system `BOOT_COMPLETED` broadcast and re-validates the action is a
   lower real-world risk than a flat HIGH implies.
3. **`UNKNOWN_ANDROID` as a missed-Gradle-findings risk.** three-times-a-day applies
   the Android Gradle plugins through **version catalog aliases**
   (`gradle/libs.versions.toml` + `alias(libs.plugins.…)`). v0.1.0 could not resolve
   those aliases to real plugin ids, so both Gradle modules were classified as
   `UNKNOWN_ANDROID`:

   - Android application modules: 0
   - Android library modules: 0
   - unknown Android modules: 2

   Rules scoped to `APPLICATION` or `LIBRARY` therefore never ran against the app
   module — a potential **false negative** (missed Gradle findings). This motivated
   the v0.1.1 plugin-alias resolution work (see below).

## Performance

- Cold end-to-end run time: **≈ 1 minute 20 seconds**.

## v0.1.1 follow-up — plugin alias resolution

The `UNKNOWN_ANDROID` risk above is addressed by teaching `GradleAndroidConfigScanner`
to resolve `[plugins]` aliases declared in the standard `gradle/libs.versions.toml`
version catalog and applied via `alias(libs.plugins.…)`.

### Effect on the three-times-a-day scan

| Metric | Before | After |
| --- | --- | --- |
| Android application modules | 0 | 1 |
| Android library modules | 0 | 0 |
| unknown Android modules | 2 | 1 |
| Gradle Android configs | 2 | 2 |
| Findings | 1 | 2 |

- The `androidApp` module (`alias(libs.plugins.androidApplication)` →
  `com.android.application`) is now correctly classified as **APPLICATION**.
- The `shared` module applies `alias(libs.plugins.androidMultiplatformLibrary)` →
  `com.android.kotlin.multiplatform.library`, which is **not** one of the four known
  Android plugin ids, so it correctly remains **UNKNOWN_ANDROID**.
- Because `androidApp` is now APPLICATION and its release build type declares
  `isMinifyEnabled = false`, the previously **dormant** application-only rule
  `gradle.release.minify_disabled` (MEDIUM) now fires. This is a **real, previously
  missed finding**, not a regression — it is exactly the false negative that
  `UNKNOWN_ANDROID` was hiding.
